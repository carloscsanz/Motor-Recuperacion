Mac Pro Review: Follow-up

by Robert Mohns

Overview

In the time since we published our Mac Pro review, we've continued to test the Mac Pro, adding more memory, experimenting with RAIDs, and testing Apple's latest update to Boot Camp software. We have also developed a few questions about Mac Pro hardware and reliability.

In the days ahead, we plan to post more updates about RAID, gaming and hardware.

Memory Performance

We bought a 1GB memory kit from Ramjet, consisting of two 512-MB FB-DIMMs, equipped with heat sinks and installed it quickly and easily, following Apple's instructions (part of which are printed inside its side cover): Turn off the Mac Pro, remove its power cord,open the side panel, pull out the empty memory riser card, insert the memory modules and re-assemble. Fast and easy.

Ramjet's FB-DIMMs have smaller heatsinks than Apple's monsters; there are no protruding cooling fins, and they are lightweight by comparison. Despite Apple's cautions that non-Apple style heatsinks can increase fan speed and noise, we didn't notice any change in the Mac Pro's volume level. (It's still too quiet for our decibel meter to register.)

In many previous Macs and PCs, memory performance has increased as much as 5 to 8% when pairing up memory modules. We wanted to see if the Mac Pro would show similar performance gains from optimally matched memory, despite FB-DIMM's exotic new serial architecture, which works very differently from traditional parallel, interleaved memory busses. (We link to more technical details from Anandtech below.)

Geekbench

We ran Geekbench 2006 as the initial test of our updated and optimized memory configuration, which requires four matched FB-DIMM modules installed in the correct locations across the two riser cards.

The Mac Pro's overall Geekbench score increased from an already-impressive 296 to 343 – nearly twice the score of the Power Mac dual-core 2.0-GHz G5 and about 16% better than the stock, 1-GB Mac Pro configuration.

As we expected, pure CPU benchmarks such as Geekbench processor emulation, encryption, JPEG compression/decompression, and fractal calculations didn't change at all - no discernable improvement in either integer or floating point operations. Presumably these tests stayed within Level 2 cache and failed to exercise the memory bus.

The Geekbench improvements all show up in its memory speed test, which should indicate effects on general application performance, since almost every real application uses far more data than fit in the processors' fast, local caches.

Memory read performance increased 10% in Geekbench – a pretty good improvement, but the real speed boost came in memory writes. Ranging from 16% to 99% faster (double the speed!), these test results really boosted the overall Geekbench score. The upgraded Mac Pro fared especially well in multi-threaded tests, although even single-threaded tests did very well, ranging from 16% to 33% faster than Apple's stock 1GB configuration.

There were a few odd exceptions, though. The bzip2 decompression test was 87% slower in the multi-threaded test, yet unchanged in the single-threaded. And the dot product floating point test was 97% slower with the extra memory!

Overall, even with the strange exceptions, these sorts of speed improvements are nothing short of astonishing. In the past, interleaved parallel memory has yielded improvements of 5% to 8%. The various memory and stream performance components of Geekbench averaged a whopping 52% speed-up.

As discussed extensively in Anandtech's technical notes, the Mac Pro's serial architecture enables Intel to increase raw throughput, and add more memory slots than traditional designs (in which, the more memory you add, the slower the entire memory system gets). But the drawback is increased latency (the time required to perform a memory read or write), which can really slow down a system.

So, how does the Mac Pro compare to a Quad G5, the reigning champion of Macintosh memory performance to date? Results were mixed.

The Mac Pro's single-threaded, sequential memory read and write scores were 101% and 131% faster (yes, more than double the speed) vs. the G5. Multi-threaded scores were 164% and 23% faster, respectively. Yet Stdlib write was 50% *slower* than the Quad G5.

Memory stream performance showed smaller differences; the Mac Pro's scores ranged from 32% faster to 27% slower than the Quad G5.

Overall, the 2GB Mac Pro memory and stream tests averaged 21% faster than the Quad G5's scores.

Photoshop

Photoshop (which runs via the Rosetta translator) appeared to benefit only slightly from the paired sets of FB-DIMMs. It got a significant boost simply from having more memory available, as we expected. The Kaleidoscope test, which uses CPU more than memory or disk, showed a fair improvement of about 15% which we thought could be due to memory speed - but after monitoring Photoshop's memory usage with the command line utility 'top', we concluded the majority of the improvement came from having more memory available, rather than from memory speed per se.

Quicktime Export

Our Quicktime tests showed only minor improvements. The MPEG-4 and H.264 exports were about 3% faster. The 3GP export test improved by about a tenth of a second (about 1%), well within the human margin of error for our handheld stopwatch. Quicktime, it appears, benefits only marginally from paired FB-DIMMs.

RAW Processing in Aperture

Aperture fared much better, showing a 16% improvement in our export test. This test exports 34 RAW images to JPEG.

(The images used were from Aperture's sample Tibet Portraits album, shot with a Nikon D2Hs.)

Doom 3

Doom 3 showed significant gains at lower resolutions, but much less at higher resolutions where the video card is the main limiting factor. At "Medium" quality (640x480), Doom 3 ran 22% faster than it did with 1GB, but only 5% faster at "High" quality, and 3% at "Ultra".

We wondered if the low-resolution speedup might be due to higher resolutions requiring more memory, but Doom 3's resident memory size is only a few hundred megabytes during the timedemo, so we think not.

Boot Camp

Apple caused quite a stir when it released Boot Camp, a utility that makes it easy to install Windows XP on an Intel Mac. Providing Windows drivers, a utility to make a Windows-compatible partition, and step-by-step instructions, Boot Camp appears to have made the Mac a viable multi-OS platform.

MacInTouch has been deluged with questions about Boot Camp's performance. Chief among the questions is how well Boot Camp runs Photoshop for Windows, from those considering using a Mac Pro or MacBook Pro as a Windows Photoshop platform until Adobe releases a Universal version next year.

We used Boot Camp to install Windows XP Home Edition (service pack 2) on our test Mac Pro. Compatibility appears good in our limited testing; the Mac was indistinguishable from any other Windows machine...

...except it was slow. Very, very slow. For example, the Kaleidoscope Photoshop test that took 20 to 30 seconds on our other machines (including a Dell 9150) took over five minutes in Boot Camp Windows!

Some research quickly turned up Boot Camp's Achilles heel: hard drive access. Windows XP does not support the modern EFI firmware used in Apple's Mac Pro, only Windows' twenty year old BIOS system. (Microsoft's latest Vista beta doesn't support EFI, either.) So, Apple wrote a compatibility layer for EFI to emulate a BIOS - the "Compatibility Support Module". The Mac Pro hardware implements very high performance disk modes known as AHCI and NCQ, but Boot Camp's BIOS emulator doesn't put the disks into this mode, instead using an older, slower mode that omits AHCI, NCQ or even direct memory access mode. This effectively cripples Boot Camp's performance.

[Thanks to Brian Williams for getting to the bottom of this. See his post in the Apple Discussion Forums, linked below, for more technical detail.]

Did Apple intentionally cripple Boot Camp? We don't know - but it does appear that the route Apple chose was easier, so there may be technical difficulties to getting Windows XP to run with the Mac Pro's modern firmware and drives at full speed.

What this means, practically speaking, is this: While Boot Camp is excellent for general compatibility purposes, it is unable to run Photoshop at any sort of competitive speed.

(We don't have a pile of PC games on hand to test, but we suspect gaming performance also may be sub-par, which would be a pity, since the Mac Pro's hardware is excellent and should make a fine PC gaming machine.)

We have recently been sent instructions for getting full SATA performance out of Windows XP under Boot Camp. When we've had a chance to test this, we'll update our Windows benchmarks and this report.

One final performance note: Assuming disk performance is fixed by Apple or is not the bottleneck for your application, those wishing to maximize Windows performance should install Windows XP Pro, because XP Home Edition only supports one processor, not both of the Mac Pro's Xeon CPUs. (However, XP Home does support both processor cores in one Mac Pro CPU or the one Core Duo CPU used in the iMac and MacBook models).

RAID

The Mac Pro's four fast internal hard drive bays practically beg for high performance RAID storage systems. But, what, exactly, is RAID?

RAID, or "Redundant Array of Inexpensive Disks", is a set of techniques for combining multiple hard drives into one logical storage volume. Depending on the implementation, RAIDs can increase storage capacity, speed, fault tolerance, or even all of these at once.

Mac OS X includes software support in the Disk Utility program for two types of RAID, striping (RAID 0) and mirroring (RAID 1).

Striping provides very high data throughput and doubles capacity but offers absolutely no redundancy of data. Statistically, RAID 0 setups will fail more frequently than a single drive. They are, however, very well suited to working with large files where speed is of critical importance. Uncompressed video editing and Photoshop scratch disks are two common applications of RAID 0. Just opening applications can be much faster with RAID 0, as programs are loaded from disk into memory.

RAID 1 – "mirroring" – provides no improvement in speed and actually halves the capacity of two drives duplicating the entire contents of one onto the other for redundancy. The failure of one drive will not cause the failure of the whole system, because the system can ignore the failured drive and keep functioning with the other until a replacement arrives. This makes it ideal for high-availability applications such as servers that hold critical data.

RAID 5, or "stripe with parity", combines high speed, high capacity and fault tolerance. Data is stored across several disks, along with "parity"error-checking information that allows the RAID to recover from the failure of one disk (or sometimes more, depending on the particulars of the RAID setup). RAID 5 is among the fastest and most robust RAID variants in use, but it is technically difficult to implement in software and is normally implemented using a dedicated hardware controller instead.

In general, hardware RAID solutions provide the best speed and OS compatibility. However, due to that extra hardware, they cost more to implement. (RAID is making its way into consumer devices, though. LaCie's BigDisk series of Firewire drives uses a hardware RAID controller to join two drives internally, presenting themselves to their host as a single device, and other similar products are also available.)

RAID on the Mac Pro

Intel's logic board chipset for the Xeon includes a number of optional components that manufacturers can use – or not – at their discretion. Among them is support for hardware RAID 0, 1 or 5 configurations, but Apple appears not to have included any of these options in the Mac Pro. Instead, RAID on the Mac Pro is implemented in software, as before, via the Disk Utility application.

Mac OS X supports mirrored or striped volumes (and even 0+1 and 1+0 combinations). There has been some confusion about RAID 0 boot support for Intel Macs. One Apple Knowledge Base article says it is not supported, and we had problems booting the Mac Pro from a RAID 0 configuration (even though the Power Mac G5 can start up from a striped RAID); however, several MacInTouch readers report success booting a Mac Pro from a RAID 0 volume.

Even if Mac OS X and the Finder cannot gain the direct benefit of RAID 0 speed on the Mac Pro, other applications and operations can. We tested several configurations:

  • A pair of mirrored Hitachi 250 GB drives
  • The same Hitachi drives striped into a 500 GB volume
  • The two Hitachis and the Apple-supplied Seagate Barracuda drive striped into a 750 GB volume (booting from a fourth SATA drive, an 80-GB Maxtor drive included originally in a Power Mac G5.)

We confirmed that Mac OS X starts from a mirrored RAID with no problems. This presents no speed gains (and a slight theoretical slowdown in write speed), but offers data redundancy.

A striped RAID is far more interesting from a performance point of view. After we moved Photoshop's scratch space to the two-disk RAID, benchmarks improved 10% to 30%. Extremely disk-intensive tests such as the RetouchArtists action showed the most improvement. When we used a three-disk RAID, they improved a bit more, though not nearly so dramatically.

Our standard file copy test demonstrated the raw throughput a RAID stripe is capable of. In stock form, the Mac Pro copied a 207.5 MB folder in 8 seconds – very good compared to a G5 equipped with a Western Digital Raptor drive, which clocked in at 7.6 seconds. With a two-disk RAID, the Mac Pro set the mark at 4.5 seconds, far better than our previous record holder (a striped RAID in a low-end Power Mac G5). With a three-disk RAID, the Mac Pro took just 2.7 seconds.

With RAID 0 and its fast SATA drive bays, the Mac Pro is a disk storage speed demon.

Limitations

Traditionally, hardware RAID has provided the ability to start a Mac from a RAID 1 or 5 volume, but this is not an option on the Mac Pro. Its elegant slide-in drive bays prevent access from the RAID PCI cards we could use with previous Power Macs. Barred from this expansion option, we would have liked to see Apple including the RAID support Intel designed into the Xeon's chipset.

While Apple also sells the Xserve RAID, and it does provide good value for the dollar as it expands, not everyone has $6,000 to spend on a RAID system, or a server rack to put it in. With 500 and 750 GB SATA drives getting cheaper all the time, we'd prefer to build our own solution from inexpensive commodity parts.

External RAID systems are an option, but Mac OS X's RAID software can be touchy when combining separate external drives (especially when any error correction delays or latencies are involved), so a hardware-based external RAID system makes a lot more sense. On the software side, though, MacInTouch readers recommend SoftRAID as a more flexible and robust solution (although full SoftRAID Mac Pro support is waiting on an operating system fix from Apple - see SoftRAID FAQ).

Overall, though, we're impressed with the performance available from the Mac Pro. With four drive bays, we can mirror our startup drive, store working data on a fast two-disk RAID, and perform regular backups to Firewire 800 drives for quick recovery on another Mac if necessary. That's more expansion options than we've had from an Apple desktop in many years.

What are all those cores for?

The Mac Pro's four processing units, or "cores," put it squarely into territory once occupied only by "big iron" servers and incredibly expensive workstations — and one long-departed Mac clone, the Daystar Genesis quad-604 machine. But what advantage do you really get from all those cores, anyway?

A word on multi-tasking, -processing and -threading

Earlier versions of Mac OS, from System 7 through Mac OS 9, supported multiprocessing in a very limited manner. Only one CPU was used normally, and that CPU did everything. The other CPUs could be accessed by specially-written programs, including Photoshop (via plug-ins), but provided no general boost to applications. The main CPU was shared by every task. When your email client was fetching mail, your word processor wasn't repaginating and your web browser stopped loading its page.

Glossary of terms

Multitasking
Running multiple programs at the same time.
Multiprocessing
Using more than one processor in a single computer.
Multithreading
Running multiple tasks within a program at the same time; does not require multiprocessing, but it helps.
Symmetric multiprocessing
Scheme allowing the operating system to allocate any available processor or core to any program at any time.
Core
A processing unit or CPU. More than one may be on a single chip, in which case it is referred to as a multi-core chip.

By switching between tasks (programs) very quickly, or "multitasking," the computer made it appear that everything was happening simultaneously. But it wasn't, really, and we could see this, because when we asked the computer to do a lot of multitasking, everything slowed down.

The Mac OS X operating system was built from the ground up for SMP, or symmetric multiprocessing, based on the Mach microkernel. Mach was developed at Carnegie Mellon by Rick Rashid, who now oversees Microsoft Research, and Avie Tevanian, who worked under Steve Jobs for years at NeXT and Apple, but left Apple earlier this year.

Mac OS X uses every available processor for the various programs it is running. When your email client fetches mail, your web browser doesn't have to stop loading a page, because the operating system (which manages tasks) allocates one program to one CPU (or core), and another program to another CPU. And it moves them from CPU to CPU as needed — no CPU is solely dedicated to just one process. (This freeform rebalancing is why it's called "symmetric" multiprocessing.)

Mac OS X also supports multithreading, which is multitasking within a single application. Multithreading is what enables Word and Mail to check spelling in the background while you type, and Safari and FireFox to load several web pages at once. Before Mac OS X, applications could manage this within themselves, but each still acted as just one program (or "process", in Unix-speak) to the system.

With Mac OS X, a program can create threads that the operating system can manage automatically — and allocate to multiple CPUs. For example, when typing in Word, a thread on CPU #1 might be handling your typing, while a separate thread on CPU#2 is checking for spelling errors. Cinebench can render multiple sections of an image simultaneously, each with its own thread and processor. What this means to us computer users is that the computer feels more responsive, because there are more resources (CPUs) working on the various tasks (threads and processes) that need to operate.

And it means that a computer can get more work done because it has more resources on the job. An analogy would be that there are two ways to ship more apples from state to state: you can get a faster truck, or more trucks. —more GHz, or more cores.

So more is better, right?

It seems simple: if one core is good, two is twice as good, and four is four times as good. Right?

Maybe.

Every modern personal computer operating system has a lot of background processes. These processes are the infrastructure of your desktop experience — managing network connections, sync services, file sharing, iTunes, menu bar icons, Dashboard, and much more. (Open Activity Monitor, found in your Utilities folder, and set its monitor window to show "All Processes". It's enlightening.) These processes are typically designed by their programmers to be fast and take few resources, but en masse, they do require CPU time.

A multiprocessing computer, such as the Dual G4/G5 Macs and Intel Core Duo-based Macs, will feel "faster" and more responsive than a single processor computer, because these background processes can be put on different CPUs than the programs you are actively using.

Most typical productivity programs, such as MS Word, Safari, and Mail, become more responsive (feel "faster") with two processors but show little additional improvement with four. They just aren't demanding enough to fully utilize the extra cores.

In our testing on the Quad Xeon Mac Pro, we found that very rarely were all four CPUs fully engaged. The computer has more computing capacity than a typical home or office user needs.

Media applications such as Photoshop, GarageBand, Final Cut and Aperture are more aggressively optimized to use the extra capacity of a quad-core machine. Media producers will see a real and measurable benefit — but our benchmarking still shows that you don't get nearly the extra mileage out of CPUs #3 and #4 that you get out of #1 and #2.

Eight core upgrades

In early September of this year, Anandtech created an eight-core Mac Pro using pre-production samples of Intel's unreleased quad-core "Clovertown" processors. They wrote, "We definitely had a difficult time stressing 8 cores in the Mac Pro, but if you have a handful of well threaded, CPU intensive tasks, then a pair of slower Clovertowns can easily outperform a pair of dual core Woodcrest based Xeons."

Unfortunately, due to non-disclosure agreements with Intel, they were unable to release hard performance numbers. For that, we had to wait for Cnet to perform a similar test with production versions of those chips in November. For highly-optimized, multithreaded tests such as Cinebench, they saw performance increase by 31% with eight cores — not even close to the 100% doubling in available CPU capacity over the four-core Mac Pro, despite all eight cores running at full capacity. And iTunes encoding and Quake 4 frame rates improved by only 10%.

Traditionally, the barrier to performance for highly multiprocessing computers has been limited I/O capacity — memory bandwidth and disk performance. It's difficult to keep even two modern CPUs fed with data at the same time. Four or eight processors using the same memory and disk rapidly come into competition for resources. The new high-bandwidth memory bus used in the Mac Pro attempts to deal with this, but we don't know how well it will scale to support eight demanding processor cores.

Core conclusions

Based on our experience living with single processor, dual-processor and quad-processor Macs, we think that a duallie is the sweet spot for price/performance. And thanks to the Intel Core Duo series, it's an affordable sweet spot.

The Quad Xeon Mac Pro is costly, but incredibly fast. Its extra capacity isn't a great value for undemanding users of email, web and word processing. But if you're working with any sort of media — graphics, sound or video — its four cores can't be beat. They really do make a difference.

There has been a lot of speculation about future eight-core Mac Pros from Apple. We might see them, but few users will really have any use for them. Until media applications are better optimized to exploit many CPUs, and memory bandwidth is improved, such an exotic machine would be better suited to a high-capacity server than to a single user's need.
